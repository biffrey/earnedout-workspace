## Run 1 — ABORTED (12:38 local, manual) — kept for the record

**Scope:** Normal nightly — all active platforms in `config/search_config.md`

### Authentication Outcome

The skill's documented auth gate is a **trial `op read`** of the canonical
1Password item (`op://Personal/dealstream.com/username`). Per
`config/credentials-setup.md` and the user's explicit run instruction, the
trial read — not `op whoami` — is what gates the run. `op whoami` was
deliberately **not** used as the gate (under the desktop-app integration
it falsely reports "not signed in" even when reads succeed; this caused
the 2026-05-23 → 06-04 dark period).

#### Probe results

| Probe | Result |
|---|---|
| `op --version` | `2.33.1` (CLI installed) |
| `op account list` | account `bb@braxton.ai` (`R22YQLTCLFCCFO6HZHL6JPZJDU`) on `my.1password.com` |
| `pgrep 1Password` | desktop app + helpers running (PIDs 97450, 87492, 97460, 97461) |
| `op read "op://Personal/dealstream.com/username"` (attempt 1, 12:38:53) | **FAILED** — `error initializing client: authorization timeout` (exit 1, empty value) |
| `op read "op://Personal/dealstream.com/username"` (attempt 2, 12:40:06) | **FAILED** — `error initializing client: authorization timeout` (exit 1, empty value) |
| `~/.config/earnedout/op.env` (service-account token fallback) | **NOT present** — only `op.env.example` exists |

The trial read itself failed (twice), so the documented stop condition
fires. No platform was queried.

### Root Cause (most likely)

The 1Password desktop app was running but did not grant the CLI
authorization handshake. The two-attempt timeout pattern matched the
classic "CLI integration toggle off" or "biometric prompt firing
invisibly" failure modes — not a missing-item or wrong-path issue
(the item path was reconciled on 2026-05-21 and the prior nightly
infra confirms it resolves when the integration grants access).

### Resolution

Run 2 (17:13 UTC, above) succeeded because the runner (`run-overnight-search.sh`)
resolved the DealStream credentials directly from the macOS login Keychain
generic-password items (`earnedout-dealstream-username` / `-password`) and
passed them to the skill via env vars, bypassing the `op read` path entirely.
The skill's updated fail-loud gate — "credentials present in env vars" rather
than "`op whoami` succeeds" — fired green and the run completed end-to-end.
